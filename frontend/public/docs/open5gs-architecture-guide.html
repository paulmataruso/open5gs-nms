Reference Guide

Understanding Open5GS: architecture, plumbing, and protocol

A from-scratch guide to how Open5GS's network functions work, how they're wired together over IP, how to lay them out across real infrastructure, and what actually happens on the wire when a phone joins the network.

4G EPC 5G SA

What is Open5GS

Open5GS is a C-language open-source implementation of the mobile core network — the part of a 4G or 5G network that sits behind the radio, handling everything from authentication to routing your phone's traffic onto the internet. It implements both a full 4G EPC (Evolved Packet Core) and a full 5G SA (Standalone) core, tracking 3GPP Release-19. It's licensed under the GNU AGPL-3.0, with commercial licensing and support available separately through NewPlane.

Both cores are built the same way: a set of independent daemons, one per network function, each its own OS process with its own YAML config file, talking to each other over the network — even when that network is just loopback on one laptop. That's the whole idea of this guide: once you understand what each daemon does and what protocol it uses to talk to its neighbors, the leap from "everything on one machine" to "spread across a real network, with an edge site 40ms away" is just moving IP addresses around — not redesigning anything.

This guide covers architecture and protocol — what each daemon does and how they're wired together. For exact YAML syntax and every config key, see the YAML Config Reference alongside it.

4G EPC vs 5G SA, at a glance

The two cores solve the same problem — authenticate a device, track where it is, and get its traffic to the internet — with different protocols and a different split of responsibilities. Open5GS runs both at once if you want, sharing the SMF and UPF between them.

Concern4G EPC5G SA

Mobility/session anchorMME (mobility) + SGW-C/PGW-C (session)AMF (mobility) + SMF (session) — the same split, renamed and SBI-native
User-plane forwardingSGW-U / PGW-U (GTP-U)UPF (GTP-U) — literally the same daemon in Open5GS, shared across both cores
Inter-NF communication styleDiameter (S6a, Gx) — point-to-point, peer knows its peer's addressSBI — HTTP/2 + JSON, NFs discover each other via NRF instead of static peer lists

Why this matters for the rest of the guide: everything from here on groups naturally into "4G-only", "5G-only", and "shared" — the same three-way split you'll see on the sidebar tags throughout.

4G EPC functions 4G

Five daemons, all Diameter- or GTP-based, none of them SBI. The MME is the hub everything else in this list reaches through.

mmeMobility Management Entity
The only core node the eNB talks to. Owns UE mobility state (which tracking area, which eNB), drives attach/detach and handover, and relays NAS signaling between the UE and the rest of the core — it never touches user-plane traffic itself. Every UE it's serving has an EMM context (EPS Mobility Management: GUTI, security context, TAI list) and, per active bearer, an ECM/EPS bearer context (which SGW/PGW, which QCI). It's the one node that has to track both at once, which is why it's the busiest signaling point in the whole 4G core.
Handles directly: Attach/Detach, Tracking Area Update, Service Request, S1/X2 handover decisions, paging (triggering it, not executing the radio side), GUTI (re)allocation, NAS security mode control.
Client toward
HSS (S6a: auth vectors, location update), SGW-C (S11: session setup)
Server for
eNB (S1AP signaling)
hssHome Subscriber Server
The subscriber system of record for 4G — stores subscriber profiles (IMSI, MSISDN, provisioned APNs, QoS profile per APN, roaming permissions) and generates authentication vectors on request using the subscriber's permanent key K. Purely a Diameter server; it never initiates a connection to anything — every operation is some other NF asking HSS a question.
Key S6a operations: Update-Location-Request/Answer (MME registers as the UE's current server), Authentication-Information-Request/Answer (fetch auth vectors), Purge-UE, Cancel-Location (network-initiated deregistration), Insert-Subscriber-Data (mid-session profile push).
Client toward
— (none; pure responder)
Server for
MME (Diameter S6a)
sgwcServing Gateway — Control
The control-plane half of the Serving Gateway, split out by Open5GS to mirror the 3GPP CUPS architecture even in 4G — it manages GTP-C session/bearer state (which TEIDs map to which UE, which QCI per bearer) and tells SGW-U what forwarding rules to install over PFCP, but forwards no user data itself. Every bearer it tracks has a pair of GTP tunnel endpoint identifiers (TEIDs) — one on the eNB side, one on the PGW/UPF side — that SGW-U actually switches packets between.
Client toward
SGW-U (PFCP session/rule push)
Server for
MME (GTP-C S11)
sgwuServing Gateway — User
The forwarding-only counterpart to SGW-C — takes PFCP instructions and moves actual GTP-U bearer packets between the eNB and the PGW-U side, on the fast path, with no signaling logic of its own. It holds no session state beyond what SGW-C hands it: a TEID-keyed forwarding table. This is deliberate — it's the one component in the chain that has to be fast, not smart, and it's the one you'd scale out first under load since it's stateless from the control plane's point of view.
Client toward
— (rule-taker, not initiator)
Server for
SGW-C (PFCP), eNB (GTP-U bearer data)
pcrfPolicy and Charging Rules Function
Decides QoS and charging policy for a session and pushes the decision down over Diameter Gx when the PGW-C (SMF, in Open5GS) asks for one at session setup. The decision itself is a set of PCC rules — traffic filters bound to a QCI (QoS Class Identifier, the 4G equivalent of 5G's 5QI) and a charging key. Can also push mid-session rule updates (Re-Auth-Request) if policy needs to change without tearing the session down.
Client toward
— (pure responder)
Server for
SMF/PGW-C (Diameter Gx)

Protocol/port specifics for each interface are covered together in the Interface Map section, so they can be cross-checked in one place rather than repeated per card.

5G SA functions 5G

Twelve daemons. Everything here is SBI (Service-Based Interface) — HTTP/2 with JSON bodies — except the AMF's radio-facing side. The defining difference from 4G: instead of static Diameter peers, every one of these registers with the NRF and discovers its peers dynamically.

amfAccess and Mobility Management Function
The gNB's only point of contact and 5G's counterpart to the MME — owns UE mobility/registration state and relays NAS signaling, but (unlike MME) reaches every other core NF over SBI instead of Diameter. Tracks each UE's 5GMM state (registered/deregistered) and connection state (idle/connected) independently — a UE can be registered but idle (no active N2 signaling connection, radio resources torn down to save power) without ever deregistering.
Handles directly: Registration, Deregistration, Service Request (idle→connected), paging orchestration, GUTI/GUAMI allocation, NAS security context (post-AKA), Xn/N2 handover decisions, N1N2MessageTransfer (piggybacking NAS onto NGAP for the gNB to deliver).
Client toward
NRF/SCP, AUSF (N12), UDM (N8), SMF (N11), PCF (N15), NSSF (N22)
Server for
gNB (N2/NGAP), peer NFs (SBI)
nrfNetwork Repository Function
The phone book every other 5G NF is registered in. There's no static config anywhere telling the AMF "the SMF is at this address" — it asks the NRF, every time, which is what makes 5G's SBI mesh dynamically discoverable instead of hand-wired. Each registrant submits an NF Profile: NF type, instance ID, the PLMN(s) and S-NSSAI(s) it serves, and its own SBI address — the NRF matches discovery queries against these fields.
Handles directly: Nnrf_NFManagement (register/update/deregister, heartbeat keepalive), Nnrf_NFDiscovery (the actual lookup query), and NF status change subscriptions so a watching NF gets notified if a peer it depends on goes down.
Client toward
— (registry root; rarely a client itself)
Server for
every other 5G NF (registration + discovery)
ausfAuthentication Server Function
A thin, mostly stateless broker: takes an authentication request from the AMF, asks the UDM for the actual authentication vectors, and hands the result back — it never talks to the UE or the gNB directly. It's the SEAF-adjacent anchor for KAUSF/KSEAF derivation in the key hierarchy (see the Security section), and decides between full 5G-AKA and EAP-AKA' based on what the subscriber's profile supports.
Client toward
NRF/SCP, UDM (N13)
Server for
AMF (SBI N12/Nausf)
udmUnified Data Management
5G's subscriber-data front end — HSS's counterpart — plus a job HSS never had: running the SIDF (Subscription Identifier De-concealing Function), which holds the Home Network private key(s) and reverses a UE's SUCI back into its real SUPI during authentication. UDM is technically a stateless front-end in the spec (UDM-FE) — the actual subscriber records live in UDR; UDM adds the crypto and business logic on top.
Client toward
NRF/SCP, UDR (actual data storage)
Server for
AUSF (N13), AMF (N8), SMF (N10) — SBI Nudm
udrUnified Data Repository
The actual database behind UDM (and PCF) — MongoDB-backed in Open5GS — exposed over SBI (Nudr) instead of a direct DB connection, so UDM and PCF never talk to Mongo themselves. Holds four broad data categories: subscription data (auth/mobility/session profiles), policy data (PCF's rule sets), exposure/application data, and structured data for AF-facing exposure.
Client toward
NRF/SCP, MongoDB (db_uri)
Server for
UDM, PCF (SBI Nudr)
pcfPolicy Control Function
PCRF's SBI-native successor — session, mobility, and UE policy decisions, queried by the AMF and SMF instead of pushed over Diameter Gx. Splits into two policy types that get requested separately: AM Policy (access/mobility — RFSP index, service-area restrictions, queried by AMF over N15) and SM Policy (session/QoS — PCC rules, queried by SMF over N7).
Client toward
NRF/SCP, UDR
Server for
AMF (N15), SMF (N7) — SBI Npcf
nssfNetwork Slice Selection Function
Answers the AMF's "which slice should this UE land on" question — and for roaming, which NSSF at the visited network to defer to. Reconciles three distinct NSSAI lists per UE: what the UE requested, what the subscriber's profile allows, and what the network actually configures — the resolved intersection determines which slice-scoped AMF/SMF instances get used.
Client toward
NRF/SCP
Server for
AMF (N22, SBI Nnssf)
bsfBinding Support Function
Remembers which PCF instance is handling a given UE/session, so a policy-facing NF elsewhere in the network can find the right PCF instead of guessing. The binding record itself is small — UE address (or SUPI) plus DNN/S-NSSAI plus which PCF owns the associated policy session — but it's the piece that makes multi-PCF deployments (and AF-triggered policy changes from outside) actually route correctly.
Client toward
NRF/SCP
Server for
PCF, other NFs needing a binding lookup (SBI Nbsf)
scpService Communication Proxy
Optional. Lets NFs address it instead of every peer directly — indirect communication — so a topology change or peer failure doesn't mean reconfiguring every NF that talks to it. Skippable entirely in a single-operator lab. The spec distinguishes two indirect-communication models: NFs can either resolve the target themselves and just route the message through the SCP, or delegate discovery to the SCP entirely and let it pick (and re-pick, on failure) the destination instance.
Client toward
NRF, next-hop SCP (multi-tier)
Server for
any NF configured to route through it
seppSecurity Edge Protection Proxy
The only NF that ever talks to another operator's core — sits at the inter-PLMN boundary, speaks N32 to the visited/home network's own SEPP, and applies topology hiding so internal NF addresses never leak to a roaming partner. N32 itself splits into N32-c (control — one-time capability negotiation/handshake between the two SEPPs) and N32-f (forwarding — the actual per-message relay, with each message re-formatted to strip internal topology details before it crosses the boundary).
Client toward
NRF/SCP, remote SEPP (N32)
Server for
remote SEPP (N32)

Shared functions 4G+5G

Two daemons Open5GS runs identically for both cores — the same running process serves 4G and 5G sessions side by side.

smfSession Management Function
The busiest config surface in Open5GS: plays PGW-C's role for 4G sessions (GTP-C to SGW-C, Diameter Gx to PCRF) and SMF's role for 5G sessions (SBI to AMF/PCF, PFCP to UPF) — out of the same process, same YAML file. For every 5G session it holds an SM context keyed by PDU Session ID, and drives that session through establishment, modification, and release by compiling policy into PFCP rules — it never touches a packet itself, only tells the UPF what to do with packets via four rule types: PDR (Packet Detection Rule — how to recognize traffic for this session), FAR (Forwarding Action Rule — where to send it), QER (QoS Enforcement Rule — rate-limit/mark it), and URR (Usage Reporting Rule — count it, for charging).
UPF selection happens here too — the DNN/TAC/Cell-ID matching covered in the Deployment section is literally the SMF choosing which UPF gets the PDR/FAR/QER/URR set for a given session.
Client toward
UPF (PFCP/N4), NRF/SCP, PCF (N7), UDM (N10), SGW-C (4G, GTP-C)
Server for
AMF (N11, SBI), SGW-C (GTP-C, 4G)
upfUser Plane Function
Where actual subscriber traffic goes. Terminates the 4G S5-U path from SGW-U and the 5G N3 path straight from the gNB, on the same forwarding engine — it has no SBI interface at all; PFCP is the only control channel it understands. Everything it does is executing the PDR/FAR/QER/URR rule set the SMF programmed into it: match a packet (PDR), decide where it goes and how to encapsulate it (FAR), apply the QoS treatment (QER), and count it for the URR's usage report back to the SMF.
Client toward
— (rule-taker from SMF)
Server for
SMF (N4/PFCP), eNB/gNB (GTP-U bearer data)
Why this matters for deployment: because SMF and UPF are the two NFs that bridge both cores, they're also the two that show up in every deployment-topology decision later in this guide — where you place the UPF is, in a real sense, where you place "the network."

Deployment scenarios

Every scenario below is the same set of daemons — nothing architecturally changes. What moves is which IP address each daemon binds to, and whether that address is 127.0.0.x or something reachable across a real network. This is the direct, practical payoff of the client/server map above.

Scenario 1 — All-in-one (Open5GS's own default)

Every daemon runs as a process on one machine, all bound to 127.0.0.x loopback aliases: MME 127.0.0.2, SGWC/SGWU 127.0.0.3/127.0.0.6, SMF 127.0.0.4, AMF 127.0.0.5, UPF 127.0.0.7, HSS/PCRF 127.0.0.8/127.0.0.9, and the 5G SBI NFs spread across 127.0.0.10–127.0.0.20. This is exactly what Open5GS ships as its quickstart config.

Fig. 1 — all 17 daemons, one host, wired over loopback only. The RAN device is the one thing that was never on loopback, even in "all-in-one" mode.

Scenario 2 — Distributed core (multi-host)

Each daemon (or a small group of them) moves to its own host, and every loopback address in its config gets replaced with a real, reachable IP. Open5GS's own guidance is explicit about two things here: the AMF's ngap.server and the UPF's gtpu.server must bind to the real LAN-facing address (they're the two interfaces a physical gNB actually has to reach), and a firewall should block a UE from ever reaching another NF's address directly — the UE only ever has a path to the gNB/eNB, never straight into the core. This is also the split a predecessor project (NextEPC, same author/architecture lineage) documented concretely years earlier, down to the specific gotcha that MongoDB itself must move off localhost to a real bind address once HSS/PCRF live on a different host from it.

Fig. 2 — same daemons, three hosts, real routable IPs between them.

The idea underneath: CUPS

Every scenario above works because Open5GS builds on CUPS — Control and User Plane Separation, a 3GPP Release-14 concept (TS 23.214) that splits SGW/PGW into a control-plane half (SGW-C/PGW-C) and a user-plane half (SGW-U/PGW-U), talking to each other over a dedicated protocol — PFCP, defined in TS 29.244, UDP port 8805. Open5GS's sgwc/sgwu pair is that split. 5G didn't need a retrofit for this: the SMF/UPF split was CUPS-native in TS 23.501 from day one — it's not that 5G "added" CUPS, it's that 5G's session architecture assumed the split from the start. This is the mechanism that makes every scenario below possible: because the user-plane half has no session logic of its own, it can be relocated anywhere reachable over PFCP without the control-plane half caring.

Scenario 3 — Combined 4G/5G core, single remote UPF

The simplest real-world split, and arguably the most common one outside a pure lab: everything stays together at one core site — both 4G control (MME, HSS, SGW-C, PCRF) and 5G control (AMF, NRF, AUSF, UDM, UDR, PCF, NSSF, BSF, SCP), plus the shared SMF — and only the user-plane pair, SGW-U and UPF, moves out to a single remote site. That remote site is typically wherever the radio itself physically lives, since the whole point is keeping the GTP-U bearer path (S1-U, N3) short and local rather than backhauling every packet to the core.

This differs from the generic "distributed core" in Scenario 2 in one important way: nothing else splits. One site to manage certs, MongoDB, and subscriber data for the whole control plane; one remote site that's pure forwarding. It also differs from the multi-edge-site Scenario 4 below in scale — this is one remote UPF, not several — which makes it the natural starting point before scaling out to more sites.

Fig. 3 — one core, one remote user-plane site. Control signaling (blue) reaches all the way back to the core no matter which generation; only bearer traffic and its internet exit (green) stay local.

To answer directly: yes — every control-plane interface (S1AP, N2, plus the PFCP session-control links N4 and Sxa) reaches back to the core site across the WAN, regardless of topology. What doesn't cross the WAN is the bearer traffic itself: the eNB/gNB hand user-plane packets to the co-located SGW-U/UPF over S1-U/N3, SGW-U forwards onward to UPF over the local S5-U hop (UPF plays the PGW-U role for 4G too, so it's the one place both generations' traffic actually converges), and UPF exits directly to the internet/data network over SGi/N6 right there at the remote site — never routed back through the core at all. That asymmetry (control always goes to the core, user-plane exits locally) is the entire mechanism CUPS is built around, and it's true of every scenario in this section, not just this one.

This is the same CUPS mechanism as every other scenario in this section, applied at its smallest useful scale — see the multi-site generalization in Scenario 4 below, and the DNN/TAC/Cell-ID selection mechanism that makes either topology configurable rather than hardcoded.

Scenario 4 — Central control-plane, multiple edge user-plane sites

Open5GS names this scenario directly: keep AMF/SMF (or MME/SGW-C) centralized, and deploy UPF (or SGW-U/PGW-U) instances out in the field near where traffic actually needs to exit — "somewhere with a high speed Internet connection" — while control functionality stays centralized. This is 3GPP's own recognized architecture too: TS 23.501 clause 6.3.3 covers UPF selection explicitly in terms of "centrally located UPF and distributed UPF located close to or at the Access Network site," and clause 5.13 covers the edge-computing mechanisms that make it work — Uplink Classifier (UL CL) and Local Area Data Network (LADN), both introduced in Release 15.

Fig. 4 — one control plane, UPF pushed out to each edge site's local breakout. Each gNB's N3 traffic never has to cross the WAN to reach its UPF.

Selecting which UPF: DNN, TAC, or Cell ID

The mechanism behind Fig. 4 isn't hypothetical — it's a real, documented field in Open5GS's own SMF config. The SMF can pick a UPF per session by matching on the requested DNN, the UE's Tracking Area Code, or even the specific cell/gNB it's attached to, evaluating entries in order and taking the first match. Note the direction of connection here too: the UPF connects to the SMF, not the other way around — so adding a new edge UPF is purely additive, nothing on the SMF side has to change except this list.

smf.yaml — pfcp.client.upf[] selection keys
smf:
  pfcp:
    client:
      upf:
        - address: 10.10.1.7      # Edge site 1's UPF
          dnn: internet
          tac: 1                  # match by Tracking Area Code
        - address: 10.10.2.7      # Edge site 2's UPF
          dnn: ims
          nr_cell_id: 0x1000001   # match by specific 5G cell (36-bit)
        - address: 10.10.3.7      # Edge site 3's UPF — fallback
          dnn: internet

This is Open5GS's canonical sample config, configs/open5gs/smf.yaml.in. TAC-based selection maps directly onto 3GPP's own selection criteria — TS 23.501 §6.3.3.3 lists "5G-AN-provided identities e.g. CellID, TAI" as inputs to UPF selection.

What actually has to cross the firewall, per scenario

Every diagram above shows the happy-path wiring — what's easy to miss is that once a link stops being loopback, it's now something a firewall rule, a routing table, and (across sites) a WAN link all have opinions about. Concretely, per scenario:

ScenarioMust be reachableMust be blockedLatency sensitivity
All-in-oneNothing — all loopbackN/ANone — same kernel, same host
Distributed coreHost A↔B↔C per Fig. 2; AMF's N2 and UPF's N3/GTP-U from the RAN subnetUE subnet → any NF address except the RAN-facing one on Host CLow — N4/PFCP control messages are infrequent per session, not per-packet
Combined core + remote UPFCore↔remote: N4 and Sxa (PFCP, UDP 8805) and the RAN's own S1AP (SCTP 36412) and N2 (SCTP 38412) reaching all the way to MME/AMF — control plane always goes to the core, regardless of generationCore's internal SBI/Diameter mesh from the remote site entirely; anything but the RAN itself from the remote site's radio-facing addressesControl plane (S1AP/N2/N4/Sxa): all tolerant of WAN latency — infrequent, per-session or per-procedure, not per-packet. Bearer traffic: never crosses the WAN — S1-U/N3 terminate locally and exit to the internet right there via SGi/N6
Central CP + edge UPFCentral↔each edge site's N4/PFCP (UDP 8805); each edge UPF's own N3/GTP-U from its local gNBEdge sites from reaching each other directly (no inter-edge traffic needed); UE subnet from the WAN entirelyN4 control: tolerant of 50–100ms WAN RTT. N3/N6 user-plane: this is exactly what you're trying to keep low by placing UPF at the edge in the first place — a central UPF instead would add that same WAN RTT to every packet, not just control messages

The asymmetry in that last row is the entire argument for edge UPF placement: PFCP is a control protocol, so it tolerates WAN latency between SMF and a remote UPF just fine — Open5GS's default heartbeatInterval-style keepalives run on the order of seconds, not milliseconds. But GTP-U is the actual user data, one packet at a time, and that's the path where round-trip latency directly becomes the user's experienced latency. A central UPF serving a far-away site adds WAN RTT to every single packet the UE sends or receives; an edge UPF only adds that same RTT to the comparatively rare N4 control exchanges.

Failure modes worth planning for

Splitting a single-host system across a real network doesn't just add IPs — it adds ways for the system to be partially broken instead of cleanly up or down, which is a qualitatively different failure mode to design against.

What failsEffectNotes
Edge site's WAN link dropsExisting PDU sessions on that edge UPF keep forwarding local traffic (UPF doesn't need the SMF link for data-plane forwarding it's already programmed for) — but no new sessions can be established, and PFCP heartbeat timeout will eventually mark the association staleThis is the direct payoff of CUPS: the user-plane rule set, once installed, doesn't need the control channel to keep working
Central AMF/SMF site fails entirelyTotal outage in the topologies shown aboveHA/redundancy (e.g. multiple AMF instances behind a load balancer, NRF-mediated failover) is a separate deployment concern this guide doesn't cover in depth
NRF (or SCP) goes downExisting NF-to-NF connections already established keep working; anything requiring a fresh discovery lookup (e.g. AMF resolving a new SMF for a first PDU session) fails until NRF/SCP returnsMotivates running NRF as the one NF you'd prioritize for redundancy even in an otherwise minimal deployment
MongoDB unreachable (HSS/UDM/UDR/PCF)New authentication and new session policy lookups fail; already-authenticated, already-connected UEs are typically unaffected until they need a fresh lookupThe NextEPC-era gotcha about binding Mongo off localhost is exactly the kind of change that turns "works in the lab" into "single point of failure in production" if it's not paired with real DB redundancy

Identifiers — IMSI, SUPI, SUCI, and friends

Every one of these answers a version of "who/where is this device," but they're deliberately not the same thing — the whole point of SUCI existing is that IMSI/SUPI must never cross the air interface in the clear.

IMSIInternational Mobile Subscriber Identity

The permanent subscriber identity baked into the SIM, used as-is in 4G. Up to 15 digits: country code, network code, then a subscriber number unique within that network.

5G's name for the same permanent identity concept — for ordinary 3GPP access it's just an IMSI-shaped value (MCC+MNC+MSIN); it only takes a different shape (a Network Access Identifier, RFC 4282-style) for non-3GPP access scenarios.

The critical rule: SUPI must never be sent over the air unencrypted. That's what SUCI exists to prevent.

Security & authentication — 5G-AKA

One shared secret, provisioned once into the SIM and the UDM, fans out through a strict key-derivation chain into a different key for every layer that needs one — so a compromise at, say, the gNB's radio-layer key doesn't expose anything upstream of it.

K
→the one long-term secret, in the SIM and in UDM — never leaves either
CK, IK
→cipher/integrity keys from the Milenage/TUAK algorithm
KAUSF

KSEAF

KAMF

KgNB
KNASint / KNASenc

Per TS 33.501 §6.2 (key hierarchy).

Milenage — the actual cryptographic algorithm

CK/IK in the hierarchy above don't come from nowhere — they're the output of Milenage, the reference AKA algorithm set defined in TS 35.205 (overview) through TS 35.208 (conformance test data), built on AES (Rijndael) as its block-cipher kernel rather than a bespoke design.

Naming collision worth flagging explicitly: one of Milenage's own input fields is called AMF — Authentication Management Field, a 16-bit operator-controlled flag set (algorithm versioning, key lifetime hints) baked into every authentication vector. This has nothing to do with the Access and Mobility Management Function network function this whole guide is otherwise about. Same three letters, completely unrelated concepts, and 3GPP's own naming is the reason for the collision, not this guide's.

Milenage takes four real inputs — K (the 128-bit permanent key), RAND (128-bit random challenge), SQN (48-bit sequence number), and the AMF field above — plus OPc, an operator-specific value derived by combining a constant OP with K through the cipher kernel, used everywhere OP itself would otherwise appear so that OP never has to be exposed on every single computation. From those, seven functions produce everything else:

FunctionOutputSizePurpose
f1MAC-A64-bitNetwork authentication code, carried inside AUTN — proves the challenge really came from the home network
f1*MAC-S64-bitRe-synchronization MAC, carried inside AUTS (see below)
f2RES / XRES64-bitThe response the UE computes and the network compares against
f3CK128-bitConfidentiality (cipher) key — feeds the key hierarchy above
f4IK128-bitIntegrity key — feeds the key hierarchy above
f5AK48-bitAnonymity key — XORed with SQN so the sequence number isn't sent in the clear
f5*AK (resync)48-bitA separate anonymity key used specifically when the UE builds a resync AUTS

Per ETSI TS 135.206, corroborated by open reference implementations (e.g. libosmocore, wpa_supplicant's Milenage module).

SQN and the AUTS resync mechanism

SQN exists purely for freshness — replay protection against a captured-and-replayed authentication vector. Home network and USIM each keep their own counter (SQNHE, SQNMS) and the USIM checks any network-supplied SQN falls within an acceptable window before accepting the challenge as fresh. When the two drift out of sync — plausible after a SIM swap between devices, or vectors generated but never used — the UE rejects the challenge with Authentication Failure (cause: SYNC_FAILURE) and returns:

AUTS construction (UE side, on resync)
AUTS = CONC || MAC-S

  where  CONC  = SQN_MS ⊕ f5*(RAND)      # the UE's own SQN, concealed
         MAC-S = f1*(SQN_MS || RAND || AMF)   # proves the UE holds K

The network verifies MAC-S, resynchronizes SQNHE to the UE's own counter, discards any stale unused vectors, fetches fresh ones, and restarts authentication from scratch.

NAS vs AS — two layers of signaling

Every message in the walkthroughs below belongs to one of two layers, and mixing them up is the single most common source of confusion when reading a 3GPP trace for the first time.

LayerRuns betweenCarriesCares about radio tech?
AS — Access StratumUE ↔ eNB/gNBRadio-specific signaling: RRC connection setup, radio bearer config, handover measurementYes — this is the radio-specific layer
NAS — Non-Access StratumUE ↔ MME/AMFMobility management (attach/registration) and session management (bearer/PDU session setup)No — identical NAS message flows regardless of eNB or gNB underneath; the RAN just transparently relays it

4G NAS is specified in TS 24.301; 5G NAS in TS 24.501. 5G NAS itself splits into two sub-protocols: 5GMM (5G Mobility Management — registration, authentication, idle/connected state) and 5GSM (5G Session Management — PDU session establish/modify/release).

Beyond initial attach — session, mobility, and teardown

Registration only gets the UE onto the network. Getting it an actual IP address, moving between cells without dropping, reconnecting from idle, and eventually leaving cleanly are all separate procedures — this is where AMF, SMF, PCF, and UPF actually do the work the earlier NF cards described.

Deregistration 5G

UE-initiated: UE sends Deregistration Request → AMF releases SM contexts with every SMF the UE has an active session with, deregisters with UDM, unsubscribes from SDM notifications, releases policy associations with PCF, and removes the UE context → AMF replies Deregistration Accept (skipped entirely if the UE indicated it's powering off). Per TS 23.502 §4.2.6.

4G Service Request & Tracking Area Update 4G

4G's equivalent mechanisms: Service Request is a UE (ECM-IDLE, with uplink data) sending a Service Request NAS message to the MME, which re-establishes S1 signaling and triggers SGW/eNB bearer setup — the downlink-triggered counterpart has the SGW buffer the first packet and signal the MME to page the UE. Tracking Area Update (TAU) is 4G's periodic "I'm still here, and possibly in a new tracking area" message: UE→MME Tracking Area Update Request (carrying GUTI + old TAI); if the MME changed since last update, old and new MME exchange UE context; MME→UE Tracking Area Update Accept (may assign a new GUTI, refreshes the T3412 periodic-TAU timer from the shared time section); UE→MME Tracking Area Update Complete only if the GUTI changed.

Per TS 23.401 (Service Request / TAU clauses).
